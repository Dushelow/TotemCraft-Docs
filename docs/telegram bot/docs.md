# Telegram‑бот для вайтлиста и заявок Minecraft‑сервера TotemCraft

## 1. Введение

Бот предназначен для автоматизации приёма заявок на регистрацию (вайтлист), обработки обращений игроков, управления тикетами и интеграции с игровым сервером Minecraft (Purpur + AuthMe). Реализован на Python с использованием библиотеки `pyTelegramBotAPI`.

**Ключевые возможности:**
- Подача заявок с валидацией ника и пароля по правилам AuthMe.
- 24‑часовой таймер повторной подачи заявки.
- Полная история заявок в CSV.
- Система тикетов и диалогов «админ ↔ игрок».
- Уведомления в Discord о новых заявках, решениях, обращениях, блокировках.
- Автоматическая регистрация аккаунта на сервере через webhook‑команду `authme register`.
- Административная панель с инлайн‑кнопками (просмотр заявок, ответы, статистика, управление).
- Ежедневное напоминание о нерассмотренных заявках.

## 3. Установка и запуск

### 3.1 Переменные окружения

Бот ожидает следующие переменные (обязательные):

```bash
BOT_TOKEN=ваш_токен_бота
ADMIN_ID=123456789               # числовой ID администратора
DISCORD_WEBHOOK_URL=https://...  # URL вебхука Discord для уведомлений
CONSOLE_WEBHOOK_URL=https://...  # URL для отправки команд на сервер Minecraft (например, вебхук, выполняющий команды через Rcon или HTTP API)
```

**Рекомендация:** для production использовать `EnvironmentFile` в systemd.

### 3.2 Зависимости

Установка через pip:

```bash
pip install pyTelegramBotAPI requests schedule pytz python-dotenv
```

### 3.3 Запуск

Скрипт `bot.py` сам запускает поллинг и планировщик в отдельном потоке:

```bash
python3 bot.py
```

**systemd‑сервис (пример):**

```ini
[Unit]
Description=TotemCraft Telegram Bot
After=network.target

[Service]
Type=simple
User=minecraft
WorkingDirectory=/home/minecraft/bot
EnvironmentFile=/home/minecraft/bot/.env
ExecStart=/usr/bin/python3 /home/minecraft/bot/bot.py
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```

## 4. Архитектура и данные

### 4.1 Файлы данных (все в одной директории с `bot.py`)

| Файл                          | Содержание                                                                 |
|-------------------------------|----------------------------------------------------------------------------|
| `pending.json`                | Активные заявки `{user_id: {nick, password, comment, tg_name, username, date}}` |
| `approved_applications.csv`   | История заявок (столбцы: Дата, TG_Username, TG_ID, Minecraft_Ник, Пароль, Статус, Комментарий_игрока, Комментарий_админа) |
| `last_application.json`       | `{user_id: ISO_timestamp}` - для 24‑часового ограничения повторной подачи |
| `blocked_users.json`          | Список `user_id` заблокированных пользователей                             |
| `chat_history.json`           | `{user_id: [ {from, text, time}, ... ]}` - до 20 сообщений на пользователя |
| `message_queue.json`          | Список `user_id` с непрочитанными сообщениями                              |
| `tickets.json`                | `{counter: int, tickets: {user_id: {id, status, message, nick, date}}}`    |
| `bot_config.json`             | `{paused: bool}` - приостановка регистрации                                |
| `admin_panel.json`            | Технический (хранит `msg_id` последнего сообщения панели, не критично)     |
| `bot_errors.log`              | Журнал ошибок с traceback                                                  |

### 4.2 Глобальные переменные состояния

```python
user_states = {}            # {user_id: {'step': str, ...}} – для многошаговых форм
admin_states = {}           # {ADMIN_ID: {'action': str, 'user_id': str, ...}}
admin_reply_to = None       # ID пользователя, с которым идёт активный диалог
registration_paused = False # флаг приостановки приёма новых заявок
pending = {}                # кеш pending.json
blocked_users = set()       # кеш blocked_users.json
active_tickets = {}         # кеш tickets.json['tickets']
ticket_counter = int        # счётчик тикетов
chat_history = defaultdict(deque)   # кеш истории сообщений
unread_messages = set()     # кеш непрочитанных
last_application = {}       # кеш last_application.json
```

### 4.3 Поток управления

Бот работает в двух параллельных потоках:
1. **Основной** – `bot.infinity_polling()` обрабатывает сообщения и колбэки.
2. **Планировщик** – `run_scheduler()` каждую минуту проверяет расписание (ежедневное напоминание в 08:00 МСК).

## 5. Модули и ключевые функции

### 5.1 Валидация (AuthMe‑совместимая)

```python
def validate_nick_authme(nick: str) -> tuple[bool, str]
```
- Длина 3–16 символов.
- Разрешены: `A-Z a-z 0-9 _`.
- Возвращает `(True, '')` или `(False, причина)`.

```python
def validate_password_authme(password: str, nick: str = None) -> tuple[bool, str]
```
- Длина 6–30, без пробелов, без кириллицы.
- Не должен совпадать с ником (регистронезависимо).
- Чёрный список слабых паролей (`WEAK_PASSWORDS`).
- Запрещены: только цифры, простые последовательности (`1234`, `abcd`), повторяющиеся блоки (`abcabc`).

### 5.2 Работа с данными

- `load_json(filename, default)` / `save_json(filename, data)`
- `read_approved_csv()` – возвращает список строк (без заголовка)
- `check_nick_already_approved(nick)` – проверка, не занят ли ник (только одобренные)
- `check_duplicate_tg_id(tg_id)` – проверяет, подавал ли этот TG ID заявку ранее (из CSV)
- `reload_pending()` – перечитывает `pending.json` (используется перед уведомлениями)

### 5.3 Отправка команд на сервер Minecraft

```python
def send_console_command(command: str) -> None
```
Отправляет HTTP POST на `CONSOLE_WEBHOOK_URL` с JSON `{"content": command}`.  
Ожидается, что вебхук на стороне сервера выполнит команду в консоли (например, через Rcon или напрямую через API панели управления).

### 5.4 Интеграция с Discord

Все функции `discord_*` формируют embed и отправляют на `DISCORD_WEBHOOK_URL`.  
**События:**
- Новая заявка
- Решение (одобрено/отклонено)
- Обращение игрока (тикет)
- Сообщение гостя
- Блокировка игрока
- Открытие/закрытие диалога
- Отмена заявки игроком
- Ежедневное напоминание о pending заявках

### 5.5 Rate limiting

```python
RATE_LIMIT = 5          # сообщений
RATE_WINDOW = 5         # секунд
RATE_COOLDOWN = 10      # секунд блокировки после превышения
```
Для администратора ограничение не действует. При срабатывании пользователю отправляется предупреждение и временная блокировка ввода.

## 6. Система заявок (вайтлист)

### 6.1 Процесс подачи (со стороны пользователя)

1. Пользователь нажимает кнопку **«📝 Подать заявку на сервер»** (инлайн‑кнопка в главном меню).
2. Проверки: регистрация не приостановлена, пользователь не заблокирован, нет активной заявки в `pending`, не прошло 24 часа с предыдущей заявки.
3. Состояние `user_states[uid] = {'step': 'nick'}`.
4. Пользователь вводит ник → валидация → сохранение → переход к `step: 'password'`.
5. Ввод пароля → валидация → переход к `step: 'comment'`.
6. Комментарий (можно пропустить кнопкой) → показ подтверждения (инлайн‑кнопки «Подтвердить» / «Отмена»).
7. После подтверждения → показ правил сервера с кнопками «Согласен» / «Не согласен».
8. При согласии: заявка сохраняется в `pending.json`, фиксируется `last_application[uid] = now`, уведомление админу и Discord.

### 6.2 Обработка заявки администратором

- Админ открывает панель → «📋 Заявки» → постраничный просмотр.
- Для каждой заявки доступны кнопки: **✅ Одобрить**, **❌ Отклонить**, **🚫 Заблокировать**, **💬 Ответить**.
- При одобрении/отклонении:
  - Если действие = `approve` – сразу отправляется `authme register <nick> <password>` через `send_console_command`.
  - Админу предлагается ввести комментарий (или пропустить). Комментарий сохраняется в CSV.
  - Заявка удаляется из `pending`, добавляется в `approved_applications.csv` со статусом.
  - Пользователь получает уведомление (при одобрении – пароль и IP сервера).
  - Отправляется embed в Discord.

### 6.3 Ограничения и защита

- **24 часа** между заявками от одного TG ID (хранится в `last_application.json`).
- Проверка уникальности ника (только среди **одобренных** заявок).
- Проверка, что TG ID ранее не подавал заявку (предупреждение админу, но не блокирует).
- При приостановке регистрации (`/pause` или кнопка в админ‑панели) новые заявки не принимаются.

## 7. Система тикетов и диалогов

### 7.1 Создание тикета

Пользователь нажимает **«🚨 Жалоба / вопрос администратору»** → выбирает:
- **«✅ Да, есть аккаунт»** → вводит игровой ник и текст обращения → создаётся тикет.
- **«❌ Нет аккаунта»** → может подать заявку (`support_new`) или отправить сообщение как гость (`support_guest`).

Тикет сохраняется в `tickets.json` с уникальным номером (`counter`).  
**Важно:** если у пользователя уже есть активный тикет, новый не создаётся – бот предлагает дождаться ответа.

### 7.2 Диалог между админом и пользователем

- Админ нажимает **«💬 Ответить»** в профиле пользователя или в уведомлении о новом сообщении.
- Устанавливается `admin_reply_to = user_id`, снимается флаг непрочитанных.
- Все сообщения админа автоматически пересылаются пользователю и наоборот.
- Сохраняется история в `chat_history.json` (до 20 сообщений).
- Админ может завершить диалог кнопкой **«❌ Завершить диалог»** или командой `/stopreply`. Пользователь может завершить диалог кнопкой **«❌ Завершить диалог»**.
- При завершении диалога закрывается соответствующий тикет (`close_ticket`).

### 7.3 Непрочитанные сообщения

- Сообщение от пользователя (вне активного диалога) добавляет `user_id` в `message_queue.json` и `unread_messages` (множество).
- Админ видит количество непрочитанных в главной панели.
- Как только админ открывает диалог или просматривает историю, сообщения считаются прочитанными и удаляются из очереди.

## 8. Административная панель

### 8.1 Основная панель

Вызывается командой `/start` для админа или кнопкой **«🏠 Главное меню»** → отображается инлайн‑клавиатура:

- **📋 Заявки** – постраничный просмотр `pending` с кнопками действий.
- **💬 Сообщения** – две вкладки: «Не отвеченные» (есть в очереди) и «Отвеченные».
- **📊 Статистика** – количество одобренных/отклонённых за сегодня и всего.
- **🔍 Поиск по нику** – ищет в `pending`, CSV и истории сообщений.
- **⚙️ Управление** – приостановка регистрации, список заблокированных, сброс таймеров, очистка статистики/диалогов, инструкция.

### 8.2 Команды для админа (текстовые)

| Команда                       | Действие                                                                |
|-------------------------------|-------------------------------------------------------------------------|
| `/pause`                      | Приостановить приём новых заявок                                        |
| `/resume`                     | Возобновить приём заявок                                                |
| `/block <user_id> [причина]`  | Заблокировать пользователя (добавить в `blocked_users.json`)           |
| `/unblock <user_id>`          | Разблокировать пользователя                                             |
| `/status`                     | Показать статус (регистрация, количество заблокированных, активный диалог) |
| `/history <user_id>`          | Показать историю сообщений с пользователем (из `chat_history.json`)     |
| `/stopreply`                  | Принудительно завершить активный диалог                                 |

### 8.3 Управление тикетами

- Админ может закрыть тикет из профиля пользователя (кнопка **«🔒 Закрыть тикет»**).
- При закрытии тикета, если диалог активен, он также завершается.
- История сообщений не удаляется, но тикет перестаёт существовать в `tickets.json`.

## 9. Планировщик задач

```python
schedule.every().day.at("08:00", "Europe/Moscow").do(daily_job)
```
`daily_job()`:
1. Перезагружает `pending.json`.
2. Если есть нерассмотренные заявки – отправляет уведомление в Discord и личное сообщение админу.

Запускается в отдельном потоке при старте бота.

## 10. Интеграция с AuthMe

**Механизм:**  
При одобрении заявки вызывается:

```python
send_console_command(f"authme register {app['nick']} {app['password']}")
```

Вебхук `CONSOLE_WEBHOOK_URL` должен принимать POST‑запросы с JSON `{"content": "команда"}` и выполнять её в консоли Minecraft‑сервера.  
**Пример реализации на стороне сервера** (например, через простой HTTP‑сервер, который отправляет команду в Rcon или через панель Pterodactyl).

## 11. Обработка ошибок и логирование

- Все исключения в обработчиках и функциях отправки перехватываются и записываются в `bot_errors.log` с помощью `log_error(e)` (которая использует `traceback.format_exc()`).
- При сбое отправки сообщения (особенно с `parse_mode`) бот автоматически повторяет отправку без форматирования.
- Ошибки при отправке команд на сервер или в Discord логируются, но не прерывают работу бота.

## 12. Расширение и кастомная доработка

### 12.1 Добавление новых команд администратора

Достаточно добавить новый обработчик:

```python
@bot.message_handler(commands=['mycommand'])
def mycommand(m):
    if m.chat.id != ADMIN_ID: return
    # логика
```

### 12.2 Изменение справочника «О сервере»

Словарь `HANDBOOK_CHAPTERS` определён в коде. Можно добавить новую главу, следуя формату:

```python
'new_key': {
    'emoji': '🆕',
    'title': 'Новый раздел',
    'text': 'Текст раздела в HTML'
}
```

и добавить ключ в `CHAPTER_ORDER`.

### 12.3 Настройка времени ежедневного напоминания

Изменить строку в `run_scheduler()`:

```python
schedule.every().day.at("08:00", "Europe/Moscow").do(daily_job)
```

### 12.4 Изменение лимитов (rate limiting, 24h timer)

Переменные в начале файла:
- `RATE_LIMIT`, `RATE_WINDOW`, `RATE_COOLDOWN`
- Логика 24‑часового таймера: `if datetime.now() - last_dt < timedelta(hours=24)`

### 12.5 Поддержка дополнительных языков

Все тексты сообщений и кнопок в коде на русском языке. Для локализации потребуется вынести строки в отдельный файл или словарь.
