## Что реально лагает и почему

Прежде чем менять что-либо в ОС, важно понять, откуда вообще берутся лаги на Minecraft-сервере.

Из личного опыта: главная причина TPS-дропов - сущности (мобы, хопперы, редстоун), а не ОС. ChunkSpawnerLimiter, VillagerLobotomizer, ClearLag и правильные настройки Paper дают куда более заметный прирост, чем любые системные твики.

Тем не менее ОС может добавлять свои проблемы поверх:

| Симптом | Причина на уровне ОС |
|--------|----------------------|
| Фриз сервера на 1–10 секунд, дроп TPS до нуля | Переполнение буфера грязных страниц, ОС блокирует Java при массовом сбросе на диск |
| Лаги при загрузке чанков, особенно при телепортации | Swap на диске (NVMe медленнее RAM в 10–100 раз) |
| `Too many open files` в консоли при 50+ игроках | Дефолтный лимит файловых дескрипторов (1024) слишком мал |
| `Connection refused` при пиковых подключениях | Маленькая очередь соединений в ядре |
| I/O-лаги при записи мира | Планировщик очереди на NVMe создаёт лишние накладные расходы |

Настройки ниже закрывают все эти проблемы.

## zRAM - сжатая подкачка в памяти

### Зачем это нужно

Когда Java-процесс Minecraft потребляет всю выделенную RAM и ОС нужно освободить память, Linux по умолчанию начинает сбрасывать страницы на диск (swap). Даже современный NVMe в 10–100 раз медленнее оперативной памяти - это прямые лаги при загрузке чанков и телепортациях.

zRAM создаёт виртуальный блочный диск прямо в RAM, но данные на нём хранятся в сжатом виде. Скорость - как у RAM, объём - немного больше благодаря сжатию. Никакого I/O на физический диск.

### Выбор параметров

Алгоритм сжатия: `lz4`

Существуют три распространённых варианта:

| Алгоритм | Скорость | Степень сжатия |
|----------|----------|----------------|
| `lzo` | высокая | средняя |
| `lz4` | очень высокая | средняя |
| `zstd` | средняя | высокая |

Для Minecraft важна минимальная задержка на сжатие/распаковку, а не максимальная плотность. `lz4` - стандартный выбор для gaming и low-latency задач. `zstd` лучше подойдёт, если RAM катастрофически мало и нужно выжать максимум объёма.

Размер: `ram / 2` (50% от RAM)

Можно поставить 70–75%, и виртуальной памяти будет больше. Но чем больше zRAM, тем выше нагрузка на CPU при сжатии/распаковке, и при высокой активности подкачки это само начинает создавать микролаги. 50% - проверенный баланс: достаточно виртуальной памяти, нет накладных расходов от агрессивного сжатия.

Приоритет: `100`

Если в системе остался старый swap-файл, приоритет 100 гарантирует, что ядро всегда сначала использует zRAM, а не диск.

### Установка

```bash
# На минимальных образах Ubuntu zram.ko может отсутствовать
# Устанавливаем extra-модули ядра на всякий случай
sudo apt install linux-modules-extra-$(uname -r)

# Если пакет не нашёлся - пробуем generic-вариант
# sudo apt install linux-modules-extra-generic

# Устанавливаем генератор zRAM
sudo apt install systemd-zram-generator -y
```

### Конфигурация

```bash
sudo nano /etc/systemd/zram-generator.conf
```

```ini
[zram0]
zram-size = ram / 2
compression-algorithm = lz4
swap-priority = 100
```

```bash
sudo systemctl daemon-reload
sudo reboot
```

### Проверка

```bash
zramctl
```

Ожидаемый вывод:
```
NAME       ALGORITHM DISKSIZE  DATA  COMPR TOTAL STREAMS MOUNTPOINT
/dev/zram0 lz4       8G        XXXM  XXM   XXM   X       [SWAP]
```

- `DISKSIZE` ≈ 8G (при 16 ГБ RAM)
- `ALGORITHM` = lz4
- `MOUNTPOINT` = `[SWAP]` - значит используется как подкачка

```bash
free -h  # убеждаемся, что swap теперь это zRAM
```

## Параметры ядра sysctl

Создаём отдельный файл конфигурации:

```bash
sudo nano /etc/sysctl.d/99-minecraft.conf
```

Вставляем все параметры (объяснения ниже):

```ini
# --- Управление памятью ---
vm.swappiness = 10
vm.vfs_cache_pressure = 50
vm.dirty_ratio = 10
vm.dirty_background_ratio = 5
vm.overcommit_memory = 0
vm.zone_reclaim_mode = 0

# --- Сеть ---
net.core.somaxconn = 4096
net.ipv4.tcp_max_syn_backlog = 4096
```

Применяем без перезагрузки:

```bash
sudo sysctl -p /etc/sysctl.d/99-minecraft.conf
```

### Управление памятью (vm.*)

#### `vm.swappiness = 10` (дефолт: 60)

Определяет, насколько агрессивно ядро сбрасывает анонимные страницы (данные процессов) в swap. Шкала 0–200: чем выше - тем раньше Linux начинает выгружать RAM на диск.

Дефолт 60 означает, что при ещё достаточно свободной RAM система уже начинает выгружать данные на диск «про запас». Для десктопов, где важна отзывчивость интерфейса, это разумно. Для сервера, где подкачка - это задержка при загрузке чанков - нет.

Значение 10 говорит ядру: «используй RAM до последнего, прибегай к swap только в крайнем случае».

> Почему не 0? При swappiness=0 на некоторых версиях ядра возникают OOM-kill'ы раньше, чем ожидается, - ядро отказывается свапировать даже когда это единственный выход. Значение 10 оставляет чуть больше пространства для манёвра.

#### `vm.vfs_cache_pressure = 50` (дефолт: 100)

Контролирует, насколько агрессивно ядро очищает VFS-кэш (метаданные файловой системы) по сравнению с page cache (содержимое файлов).

VFS-кэш технически состоит из двух частей:
- dentry cache - кэш путей и имён файлов («где лежит файл X»)
- inode cache - кэш метаданных файлов (размер, права, владелец, timestamps)

При дефолте 100 ядро чистит оба кэша с одинаковым приоритетом. При значении 50 - система сильнее бережёт VFS-кэш, реже его вытесняет.

Почему это важно для Minecraft:

Сервер работает с тысячами мелких файлов - чанки, NBT-данные игроков, конфиги плагинов. При каждом обращении к файлу ядро сначала проверяет, есть ли путь к нему в dentry cache. Если нет - идёт на диск за метаданными. При 50 эти данные дольше остаются в памяти, что снижает количество metadata I/O при интенсивной работе с чанками.

Компромисс: при агрессивном сбережении VFS-кэша page cache может чуть уменьшиться - то есть сами данные чанков иногда придётся перечитывать с диска. Для большинства серверов, где активная зона - один общий регион, это незаметно.

#### `vm.dirty_ratio = 10` и `vm.dirty_background_ratio = 5` (дефолты: 20 и 10)

Это самая важная пара параметров для борьбы с внезапными фризами при записи на диск.

Что такое «грязные» страницы (dirty pages)?

Когда Minecraft изменяет чанк, лог или NBT-данные - данные сначала записываются в RAM (page cache), а не сразу на диск. Такие изменённые, но ещё не сброшенные на диск данные называются «грязными».

Как работают оба параметра:

`vm.dirty_background_ratio = 5` - фоновый порог. Когда объём грязных страниц достигает 5% RAM, ядро запускает фоновый поток `pdflush/kworker`, который начинает плавно сбрасывать данные на диск. Процессы при этом не блокируются, сервер продолжает работать.

`vm.dirty_ratio = 10` - жёсткий потолок. Если грязных страниц накопилось больше 10% RAM и фоновый сброс не успевает, ядро начинает блокировать процессы, которые пытаются создать ещё больше грязи - то есть Java - пока часть данных не уйдёт на диск.

Именно это и вызывает классический фриз на Minecraft-сервере:

```
[Server thread] WARN: Can't keep up! Did the system time change, 
or is the server overloaded? Running X ticks behind!
```

TPS роняет до 0–5, инвентарь не открывается, чанки не грузятся, команды зависают.

Почему дефолты (20/10) хуже для Minecraft:

При дефолте фоновый сброс начинается поздно (при 10% RAM ≈ 1.6 ГБ грязи на 16 ГБ), а жёсткий потолок - высокий (20% ≈ 3.2 ГБ). К моменту, когда начинается фоновый сброс, грязи уже много, и диск работает с большими пиковыми всплесками.

Что дают значения 5/10:

- Фоновый сброс начинается при ~750 МБ грязных данных (5% от 15 ГБ) - почти постоянно, маленькими порциями
- Жёсткий потолок - ~1.5 ГБ (10%) вместо 3.2 ГБ
- Диск работает ровнее, меньше пиков, фризы короче или исчезают совсем

> Когда это не поможет: если диск объективно не успевает записывать данные даже при плавной нагрузке (бюджетный SSD с маленьким SLC-кэшем, или хостинг с перегруженным общим дисковым массивом). Это можно проверить:
> ```bash
> # Сколько грязи накоплено прямо сейчас
> watch -n 1 "grep -E 'Dirty|Writeback' /proc/meminfo"
>
> # Загрузка диска
> iostat -x 1 5
>
> # Что именно грузит диск
> sudo iotop -o
> ```
> Если диск загружен на 90–100% при активном сервере, а dirty-объём не падает - проблема в диске, а не в параметрах.

#### `vm.overcommit_memory = 0` (дефолт)

Параметр определяет, как ядро реагирует на запросы памяти, превышающие физически доступную:

| Значение | Поведение |
|----------|-----------|
| `0` | Эвристика: разрешает почти всегда, отказывает при явно безумных запросах |
| `1` | Разрешает всё без ограничений, OOM-killer убьёт процесс при нехватке |
| `2` | Строгая проверка, часто отказывает даже обоснованным запросам |

Дефолт (0) - оптимален для выделенного VPS: Java почти всегда получает память, и нет риска внезапного убийства процесса сервера в разгар игровой сессии.

Значение `1` иногда советуют «для Java», но это значит, что при нехватке памяти OOM-killer просто убьёт server.jar без предупреждения. Значение `2` может помешать запуску сервера.

#### `vm.zone_reclaim_mode = 0` (дефолт)

На многосокетных серверах (NUMA - Non-Uniform Memory Access) у каждого CPU-сокета есть своя «локальная» RAM. При `zone_reclaim_mode = 1` ядро пытается освобождать память именно в локальной зоне перед тем, как взять данные с другого сокета.

На однососкетном сервере (большинство VPS, включая Hetzner CCX) это только добавляет лишние операции. При `0` ядро берёт память там, где удобнее и быстрее.

### Сетевые параметры (net.*)

Эти параметры актуальны при 20+ онлайне, когда начинаются проблемы типа `connection timed out` или потери при подключении.

#### `net.core.somaxconn = 4096` (дефолт: 128)

Максимальная длина очереди уже принятых TCP-соединений, ожидающих обработки приложением (Java). Дефолт 128 - реликт эпохи, когда сервера не имели тысяч соединений. При пиковом онлайне - массовом заходе игроков после ребута - очередь может переполниться, и новые подключения получают `connection refused`.

4096 - разумный запас. Сами слоты занимают ~2–4 МБ RAM и никак не влияют на производительность при нормальной нагрузке.

#### `net.ipv4.tcp_max_syn_backlog = 4096` (дефолт: ~1024)

Аналогичный параметр, но для более ранней стадии - очереди SYN-пакетов (первый шаг TCP-рукопожатия). Если клиент отправил SYN, но сервер ещё не успел ответить - соединение висит в этой очереди. При переполнении - отброс пакета, повторная попытка с задержкой.

Оба параметра работают в паре и должны быть выставлены одновременно.

## Лимит открытых файлов (nofile)

```bash
sudo nano /etc/security/limits.conf
```

Добавляем в конец файла:

```
minecraft soft nofile 65535
minecraft hard nofile 65535
```

### Зачем это нужно

Linux по умолчанию ограничивает количество файловых дескрипторов на процесс - как правило, 1024. Каждый открытый файл, сокет, pipe - это дескриптор. Minecraft при работе держит открытыми:

- файлы регионов чанков (`.mca`)
- файлы плагинов, логов, конфигов
- TCP-соединения каждого игрока
- внутренние pipe между потоками

При 50+ онлайна и десятках плагинов дефолтный лимит легко исчерпывается. Результат - ошибка в консоли:

```
java.io.IOException: Too many open files
```

После которой начинают отваливаться игроки, переставать работать плагины и сыпаться NPE.

### Проверка после применения

```bash
su - minecraft
ulimit -n
# Должно быть 65535
exit
```

> Важно: изменение limits.conf применяется только при следующем входе пользователя. Если сервер уже запущен - нужно перезапустить его (или сессию).

## I/O Scheduler для NVMe

> ⚠️ Только для физического NVMe-диска. На VPS с виртуальными дисками (virtio, emulated SATA) этот шаг не имеет смысла и может быть вреден.

### Что такое планировщик I/O и зачем его отключать

Когда несколько процессов одновременно обращаются к диску, ядро ставит их запросы в очередь и сортирует - это и есть I/O scheduler. Классические алгоритмы (CFQ, deadline, BFQ) были созданы для HDD, где физическое положение головки диска влияет на порядок чтения и оптимизация очереди давала прирост.

На современных NVMe это контрпродуктивно:
1. NVMe имеет собственный аппаратный планировщик с очередями глубиной до 64K команд
2. Время произвольного доступа к любой ячейке NVMe - одинаковое (нет механики)
3. Программный scheduler только добавляет задержку на перестановку запросов, которая никому не нужна

Режим `none` отключает программную очередь - запросы идут напрямую в драйвер NVMe.

### Проверка текущего планировщика

```bash
nvme list  # узнаём имя устройства (обычно nvme0n1)
cat /sys/block/nvme0n1/queue/scheduler
# Например: [mq-deadline] kyber none
# В скобках - активный
```

### Постоянная настройка через udev

Простое `echo none > /sys/block/...` не переживёт перезагрузку. Создаём udev-правило:

```bash
sudo nano /etc/udev/rules.d/60-nvme-scheduler.rules
```

```
ACTION=="add|change", KERNEL=="nvme[0-9]*n[0-9]*", ATTR{queue/scheduler}="none"
```

Это правило применяется ко всем NVMe-устройствам автоматически при загрузке.

```bash
sudo reboot

# Проверяем после перезагрузки
cat /sys/block/nvme0n1/queue/scheduler
# Должно быть: [none]
```

## Transparent HugePages для Java

### Что такое страницы памяти и зачем их укрупнять

Процессор не работает с памятью побайтово - он берёт её кусками, которые называются страницы. По умолчанию каждая страница = 4 КБ.

У процессора есть специальный кэш - TLB (Translation Lookaside Buffer) - который хранит «карту» того, где какая страница памяти физически лежит. Этот кэш маленький: в нём помещается несколько тысяч записей максимум.

Java с heap в 10 ГБ создаёт примерно 2.5 миллиона страниц по 4 КБ. TLB физически не может всё это держать, постоянно промахивается и вынужден каждый раз пересчитывать адреса через таблицу страниц в RAM. Это называется TLB miss - скрытые накладные расходы на каждую операцию с памятью.

HugePages делает страницы по 2 МБ вместо 4 КБ. Те же 10 ГБ heap = ~5000 страниц вместо 2.5 миллиона. TLB справляется легко, промахов почти нет, CPU тратит меньше времени на трансляцию адресов и больше - на полезную работу.

На практике это даёт 5–15% снижение нагрузки на CPU при активной работе GC и загрузке чанков - особенно заметно при большом онлайне.

### Два варианта: простой и сложный

Explicit HugePages - ты заранее выделяешь фиксированный пул страниц под Java, прописываешь это в JVM-флагах. Максимальный эффект, но нужно точно считать размер, и если ошиблся - сервер не запустится. Это для тех, кто хочет выжать всё до последнего.

Transparent HugePages (THP) - ядро само решает, когда использовать большие страницы, без твоего участия. Чуть меньше эффекта, но настраивается одной командой и ничего не сломает.

Для большинства серверов Transparent HugePages - правильный выбор.

### Режимы THP

| Режим | Поведение |
|-------|-----------|
| `always` | Ядро везде пытается использовать HugePages, даже где не просят. Может давать лаги при аллокации |
| `madvise` | HugePages используются только там, где процесс явно попросил. Java с Aikar's flags умеет это делать |
| `never` | HugePages отключены полностью |

Режим `madvise` - оптимален: Java получает большие страницы когда надо, остальные процессы не затрагиваются.

### Настройка

Проверяем текущий режим:

```bash
cat /sys/kernel/mm/transparent_hugepage/enabled
# Например: always [madvise] never
# В скобках - активный режим
```

Включаем `madvise`:

```bash
echo madvise | sudo tee /sys/kernel/mm/transparent_hugepage/enabled
```

Чтобы не слетало после перезагрузки - добавляем в sysctl-конфиг:

```bash
sudo nano /etc/sysctl.d/99-minecraft.conf
```

Добавляем строку в конец файла:

```ini
# --- Transparent HugePages ---
kernel.mm.transparent_hugepage.enabled = madvise
```

Применяем:

```bash
sudo sysctl -p /etc/sysctl.d/99-minecraft.conf
```

### Проверка

```bash
cat /sys/kernel/mm/transparent_hugepage/enabled
# Должно быть: always [madvise] never

# Сколько HugePages реально используется (после запуска сервера)
grep -i hugepage /proc/meminfo
```

После запуска сервера в `/proc/meminfo` должна расти строка `AnonHugePages` - это Java использует большие страницы.

> Важно: эффект появляется только после запуска Java-процесса, не сразу после применения настройки.

## Итог и чеклист

Все описанные настройки - не магия и не «ускоряют Minecraft в 2 раза». Это устранение системных ограничений, которые мешают серверу работать стабильно.

| Что настраиваем | Проблема, которую решаем | Заметность эффекта |
|-----------------|--------------------------|-------------------|
| zRAM (lz4, 50%) | Лаги при подкачке на диск | Высокая при нехватке RAM |
| `vm.swappiness=10` | Преждевременный swap | Средняя |
| `vm.vfs_cache_pressure=50` | Частые metadata I/O | Низкая-средняя |
| `vm.dirty_ratio=10/5` | Фризы при записи на диск | Высокая при медленном I/O |
| `nofile=65535` | `Too many open files` | Критично при 50+ онлайне |
| `somaxconn=4096` | `connection refused` при пике | Средняя при 20+ онлайне |
| `I/O scheduler=none` | Накладные расходы планировщика | Низкая-средняя на NVMe |
| Transparent HugePages (madvise) | TLB miss при работе GC | Средняя-высокая при большом heap |

### Быстрый чеклист применения

```bash
# 1. zRAM
sudo apt install linux-modules-extra-$(uname -r) systemd-zram-generator -y
sudo nano /etc/systemd/zram-generator.conf
# [zram0] / zram-size = ram / 2 / compression-algorithm = lz4 / swap-priority = 100

# 2. sysctl
sudo nano /etc/sysctl.d/99-minecraft.conf
sudo sysctl -p /etc/sysctl.d/99-minecraft.conf

# 3. nofile
sudo nano /etc/security/limits.conf
# minecraft soft nofile 65535
# minecraft hard nofile 65535

# 4. I/O scheduler (только физический NVMe)
sudo nano /etc/udev/rules.d/60-nvme-scheduler.rules

# 5. Transparent HugePages
echo madvise | sudo tee /sys/kernel/mm/transparent_hugepage/enabled
# + добавить kernel.mm.transparent_hugepage.enabled = madvise в 99-minecraft.conf

# Перезагрузка для применения zRAM и udev
sudo reboot

# Проверка
zramctl && free -h
ulimit -n  # под пользователем minecraft
cat /sys/block/nvme0n1/queue/scheduler
cat /sys/kernel/mm/transparent_hugepage/enabled  # должно быть [madvise]
```

> Главный вывод: настройки ОС - обязательная база, но не замена оптимизации самого сервера. Ограничение сущностей, правильные флаги запуска JVM (Aikar's flags) и настройки Paper/Purpur дают несравнимо больший прирост TPS. ОС-твики убирают системные артефакты - фризы, ошибки, нестабильность.
