# Шаг 3 Выкачиваю файлы со старого хоста (ограничены права) на новый (в котором полный root доступ)

#### 3.1 Вариант А - через SSH (используем rsync):

	rsync -avzP -e ssh temp_user@temp_ip:/ /home/minecraft/server/

**Объяснение команд**:
   - `-a`: Архивный режим (сохраняет права, владельцев, симлинки).
   - `-v`: Verbose (показывает прогресс для больших файлов).
   - `-z`: Сжатие (экономит траффик, полезно для больших миров).
   - `-e ssh`: Через SSH.
   - `--progress`: Прогресс каждого файла
   - `--partial`: Сохраняет частично скопированные файлы (не удаляет их при обрыве)
   - `-P`: Просто объединение двух команд `--progress` и `--partial` в одну

#### 3.2 Вариант Б - через SFTP без shell-доступа к временному хостингу

1. Запустить скрин для выполнения задачи даже с закрытой консолью на компьютере:

```
screen -S transfer
```

2. **Ввести быструю команду (одной строкой):**

```
lftp -u логин,пароль sftp://host:порт -e " set net:connection-limit 50; set net:max-retries 3; set net:timeout 15; set sftp:max-packets-in-flight 128; set ftp:use-mdtm off; set cache:enable no; mirror --parallel=50 \ --use-pget-n=1 \ --only-missing \ --ignore-time \ --ignore-size \ --verbose \ --no-perms --no-umask \ / /home/minecraft/server/; quit"
```

3. **Объяснение команд:**
	- **lftp** Сама программа - продвинутый ftp/sftp/http-клиент с поддержкой скриптов и зеркалирования.
	- **-u логин,пароль** Указание имени пользователя и пароля прямо в командной строке (небезопасно, но удобно для скриптов).
	- **sftp://host:порт** Протокол SFTP + адрес сервера и порт (обычно 22, но можно указать любой).
	- **-e " … ; quit"** Выполнить указанные команды внутри lftp и потом выйти.
	- **set net:connection-limit 50** Максимальное количество одновременных соединений к серверу (очень агрессивно, обычно 5–20 хватает).
	- **set net:max-retries 3** Сколько раз пытаться повторить неудачное соединение/операцию (по умолчанию часто больше).
	- **set net:timeout 15** Таймаут на установку соединения и на ожидание ответа (в секундах). 15 сек - довольно строго.
	- **set sftp:max-packets-in-flight 128** Сколько SFTP-пакетов можно отправить без ожидания ответа (очень высокое значение → ускоряет при хорошем соединении и низком пинге).
	- **set ftp:use-mdtm off** Отключает команду MDTM (получение точного времени модификации файла). Полезно, когда сервер её не поддерживает или она глючит.
	- **set cache:enable no** Отключает кэш каталогов → lftp каждый раз будет заново запрашивать список файлов (важно при --only-missing / --ignore-*).
	- **mirror** Команда синхронизации/зеркалирования директорий.
	- **--parallel=50** Сколько файлов зеркалить одновременно (очень много → сильно нагружает сервер и сеть).
	- **--use-pget-n=1** Сколько частей разбивать **каждый** файл при скачивании (здесь 1 = не разбивать, обычная загрузка файла).
	- **--only-missing** Скачивать только те файлы, которых нет на приёмной стороне (не перезаписывать существующие).
	- **--ignore-time** Игнорировать время модификации файлов при сравнении (важно при загрузке на многие серверы, где нельзя выставить время).
	- **--ignore-size** Игнорировать размер файла при сравнении → если файл есть - он считается одинаковым, даже если размеры разные.
	- **--verbose** Подробный вывод - показывать, что именно сейчас копируется.
	- **--no-perms** Не пытаться устанавливать права доступа (chmod) на файлы/папки на приёмной стороне.
	- **--no-umask** Не применять umask при создании файлов/папок (создавать с правами сервера по умолчанию).
	- **/ /home/minecraft/server/** Откуда и куда зеркалить: / → корень удалённого сервера /home/minecraft/server/ → локальная папка
	- **quit** Выйти из lftp после выполнения mirror.
	
	Кратко - для чего вся эта комбинация
	
	Очень агрессивная, быстрая односторонняя синхронизация **только отсутствующих** файлов с сервера в локальную папку Minecraft, без проверки времени и размера, без изменения прав, с большим количеством параллельных соединений и без кэша.

#### 3.3 Проверка наличия и размеров новых файлов:

- На сервере куда качали (основной)
	`ls /home/minecraft/server/` - наличие файлов
	*Должны быть jar, plugins, worlds и т.д.* 
	 
	`du -sm /home/minecraft/server` - размер файлов в мегабайтах

- На дополнительном откуда скачивали (временный)
	`lftp -u логин,пароль sftp://host:порт`
	 
	`du -sm /`

#### 3.4 Поменять права на файлах:
	`sudo chown -R minecraft:minecraft /home/minecraft/server`

*Эта команда говорит системе: «Всё, что внутри /home/minecraft/server - теперь полностью принадлежит пользователю minecraft и его группе. Он может всё там читать, писать и удалять, а остальные - нет (если не настроены специальные права)». Без нее возможна ошибка "Permission denied" при работе с другого пользователя*

```
java -version
```

# Шаг 5 Оптимизирую операционную систему

#### I. Делаю сжатый раздел оперативной памяти через zram

- Использую технологию zRam - lz4 - 50% ставлю, т.к. 70% может привести к небольшим лагам, хоть и будет чуть больше виртуальной оперативной памяти. Но для его работы применяю утилиту zram-generator
- Его плюс в том, что он избегает I/O-лагов на NVMe, лучше для загрузки чанков, в отличии от файла подкачки на медленном диске (память в разы быстрее чем диск и если нехватку памяти компенсировать подкачкой на диске, то будут лаги на сервере в виде задержек загруки и отклика)
- К общему сведению: lz4 - стандарт для gaming/low-latency

#### P.S. На некоторых образах Ubuntu устанавливаемых на хостингах стоит минимальная сборка без лишних модулей, в том числе в которых лежит zram.ko.

#### Решение- установить пакет с extra-модулями:
```
sudo apt update
sudo apt install linux-modules-extra-$(uname -r)
```

- Если пакет не найден → попробуй варианты:
```
sudo apt install linux-modules-extra-generic
```
- или
```
sudo apt install linux-modules-extra-virtual
```

- Перезагрузи сервер:
```
sudo reboot
```

- После перезагрузки проверь:
```
lsmod | grep zram
```
- *(должен показать zram, если загружен)*

#### Далее стандартная процедура:

**Устанавливаю утилиту:**
```
sudo apt install systemd-zram-generator -y
```

**Создаю конфиг:**
```
sudo nano /etc/systemd/zram-generator.conf
```

- и ставляю в него параметры:
```
[zram0]
zram-size = ram / 2
compression-algorithm = lz4
swap-priority = 100
```
- *нажимаю Ctrl+O → Enter (сохранить) → Ctrl+X (выход).*

**Применяю и перезагружаю (обязательно):**

	sudo systemctl daemon-reload
	sudo reboot

**Проверяю, что конфиг применен:**

	zramctl

- *должно быть что-то вроде:* 
	`NAME       ALGORITHM DISKSIZE  DATA  COMPR TOTAL STREAMS MOUNTPOINT`
	`/dev/zram0 lz4       8G        XXXM  XXM   XXM   X       [SWAP]`

- объяснение значений:
	- DISKSIZE ≈ 8G (размер сжатого раздела памяти)
	- ALGORITHM = lz4 (алгоритм сжатия)
	- [SWAP] в MOUNTPOINT (значит используется как своп)


**Командой проверяю файл подкачки в системе: (должен быть zRAM в swap).**

	free -h

#### II. Оптимизация sysctl для Minecraft-сервера на Ubuntu

- Затрону две области:
	- а) управления памятью (vm.)
	- б) сетевые настройки (net.).

- *Эти параметры настраиваются через sysctl
- *Их настройка нужна чтобы уменьшить задержки, улучшить обработку соединений и оптимизировать использование оперативной памяти*

#### а) Управление памятью (vm.)

- Эти настройки влияют на то, как система использует RAM, кэш и swap. Цель - минимизировать задержки от подкачки данных на диск
- Проблемы с памятью могут проявляться в виде задержки генерации чанков *(да, такое и от пинга может быть или патери пакетов, но неправильное использование подкачки, кэша, памяти и диска можетс статьй одной из причин проблемы с загрузкой чанков и фризов чата/инвентаря/действий на сервере)*

	`vm.swappiness=10` (дефолтное значение 60)
	
	- Определяет насколько агрессивно система использует файл подкачки на диске.
	- Рекомендовано значение 10, так как это делает систему менее склонной к использованию файла подкачки.
	- Возможные значения от 0 до 200: чем выше, тем чаще система сбрасывает данные из RAM в swap-файл (т.е. файл подкачки)
	- Дефолтное значение в системе: 60

	`vm.vfs_cache_pressure=50` (дефолтное значение 100)
	
	- Контролирует, насколько агрессивно система очищает кэш и приоритет категорий файлов в этом кэше.
	- **Выделяют две категории кэша, условно:**
		- а) page cache (кэш страниц)
		- б) vfs cache (кэш метаданных / кэш структуры файловой системы)
		  
		  **Однако на самом деле vfs cache технически состоит из двух отдельных кэшей:**
			- dentry cache (dcache) - кэш имен файлов, путей, свзяей «папка → файл»
			- inode cache (icache) - кэш метаданных файла (размер, права, владелец, timestamps и т.д.)
	- Что лежит в этих категориях кэша:
		- а) page cache - настоящие кусочки файлов: картинки чанков Minecraft, текстуры, звуки, куски мира и т.д.
		- б) vfs cache - список файлов и папок где лежит каждый файл
	- По умолчанию Linux считает, что оба склада одинаково важны, однако если мы поставим 50, то:
		- система начинает сильнее беречь vfs cache
		- она меньше и медленнее его чистит
		- но может урезать page cache
	- Почему это полезно:
		- Linux дольше держит в памяти «список всех файлов и папок»
		- Серверу почти всегда хватает уже готовой информации - он почти не ходит на диск за метаданными
		- Минус в том, что настоящий кэш данных page cache (чанки, блоки) может быть чуть меньше - тогда иногда всё-таки придётся читать с SSD куски мира
	- Почему стоит его поставить:
		- Для большинства серверов Minecraft это выгодный компромисс, потому что:
			- а) дисковые операции по метаданным (проверка существования файлов) случаются очень часто
			- б) чтение самих чанков происходит реже (особенно если много игроков в одном регионе)

	`vm.dirty_ratio=10` (дефолтное 20)
	`vm.dirty_background_ratio=5` (дефолтное 10)
	
	- Эти параметры буквально переводятся как "грязные" (dirty) - ими называют измененные данные в RAM (чанки Minecraft, логи, сохранения), которые ещё не записаны на диск.
	- Параметры контролируют:
		- `vm.dirty_ratio=20` - это значит что:
			- а) если больше 20% оперативы занято грязными изменениями, то блокируются только процессы, которые пытаются создать еще больше грязи (т.е. новые записи/изменения файлов), пока не сбросится часть старой грязи на диск.
			- б) при этом не останавливается чтение с диска и другие операции (например, запуск программ и сброс старой грязи)
			- в) на практике выглядит как внезапный фриз:
				- дроп TPS (в т.ч. до 0 или 1-5) на секунды/десятки секунд
				- инвентарь не открывается и чанк не грузится, персонаж не двигается
				- команды выполняются с задержкой или зависают
				  
			- г) если грязи копится больше 20% - значит что диск записывает данные со скоростью ниже, чем эта информация в буфер грязи поступает, в итоге она переполняется. Может быть вызвано:
				- Minecraft генерит больше мелких операций (логи, изменение чанков, инвентари), чем диск может обработать
				- Даже свежий SSD может не успевать обрабатывать инфу, если он бюджетный (низкий SLC-кэш, слабый контроллер) или хостинг перенагружен пользователями (диск делят 10–20 серверов
				  
				*Как проверить: посмотреть, если во время  диск загружен на 90–100%, а количество грязных файлов высокое и не падает быстро - значит это диск тормозит всю систему.*
				
				**Команды для проверки:**
				- сколько грязи в памяти:
					`watch -n 1 "grep -E 'Dirty|Writeback' /proc/meminfo"`
					
				- загрузка диска:
					`iostat -x 1 5`
					
				- кто именно жрёт диск:
					`sudo iotop -o`
		- `vm.dirty_background_ratio = 10` - это значит что:
			- фоновая запись на диск начинается когда накопилось уже 10% грязи в оперативе
			  
			 *Есть нюанс: сброс итак работает автоматически по таймеру (каждые 5 сек проверка, и если грязные файлы старше 30 сек - они сбрасываются на диск)*
	- Дефолтные значения так и стоят на 20 и 10, как в примере
	- Я поставил 5 и 10:
		- `vm.dirty_background_ratio = 5` - фоновая запись начинается очень рано (при 5% RAM), идёт почти постоянно, но маленькими порциями, плавно (≈750 МБ при 15 ГБ RAM).
		- `vm.dirty_ratio = 10` - жесткий потолок 10% RAM: если больше - процессы тормозятся и сами срочно сбрасывают данные на диск, чтобы не было долгих зависаний (≈1.5 ГБ).

	`vm.overcommit_memory=0` (дефолт)
	
	- Параметр ядра Linux: как ОС решает, давать ли процессу (например Java) виртуальную память "про запас" (overcommit):
		- значение 0 (дефолт) = сбалансированный режим. 
		  
		  *Дает процессу память почти всегда, если не совсем безумный сценарий, когда процесс запрашивает в разы больше, чем есть RAM и подкачки.*
		  
		  *Оно лучше для выделенного VPS, так как нет риска внезапного убийства процесса сервера в неподходящий момент. Но с ним может не запуститься server.jar на старте в редких случаях, когда памяти в принципе мало.*
		  
		- значение 1 = разрешает джаве использовать больше памяти, но в случае перегрузки памяти просто убивает процесс (чаще Java)
		- значение 2 = строго проверяет запасы памяти и часто отказывает процессу

	`vm.zone_reclaim_mode=0` (дефолт)
	
	- Это настройка ядра Linux, которая заставляет систему стараться освобождать память именно на том же (сокете CPU), где она нужна, вместо того чтобы сразу брать с другого, более медленного узла.
	- У меня значение 0 (по умолчанию) - то есть значит система берёт память там, где проще и быстрее всего доступна, без лишних операций по очистке. На моем сервере (Hetzner CCX23, один сокет) это нужный вариант, т.к. у нас один сокет.
	
	
	 ***Небольшой комментарий про сокеты:***
	 - *На домашних компах всегда один сокет - один процессор. А на серверах (особенно в дата-центрах, облаках, суперкомпьютерах) - запросто два, четыре и даже больше сокетов в одной машине. Как два мозга в одном черепе.*
	 - *В нашем случае тариф CCX23 - это сервер с одним процессором (одним сокетом) в плате*

#### б) Сетевые параметры (net.)

 - Эти настройки оптимизируют TCP-соединения для множества игроков
 - Менять только если при 20+ онлайна часто возникают проблемы с упоминанием "timeout"
   
	`net.core.somaxconn=4096` (дефолтное значение 128)
	
	- Максимальное количество соединений в очереди на прослушивание (backlog для listen()). Простыми словами это максимальная длина очереди игроков которые пытаюьтся подключиться, но еще не зашли на сервер.
	
	- ОС просто держит обработанных, но подключающихся игроков в очереди, чтобы не возникало сообщения "connection refused" и люди не спамяли переподключениями.
	  
	- Этот буфер почти не жрет CPU или RAM (4096 слотов - это максимум ~2-4 МБ памяти)

	`net.ipv4.tcp_max_syn_backlog=4096` (дефолт)
	
	- Максимальная очередь для SYN-пакетов (первый запрос на соединение).
	  
	- Аналогично somaxconn - увеличение значения позволяет больше подключений, снижая задержки при пиковой нагрузке.


Как их интегрировать:

1. Открыть конфиг:
```
sudo nano /etc/sysctl.d/99-minecraft.conf
```

2. Вставить следующие параметры:
```
vm.swappiness=10
vm.vfs_cache_pressure=50
vm.dirty_ratio=10
vm.dirty_background_ratio=5
vm.overcommit_memory=0
net.core.somaxconn=4096
net.ipv4.tcp_max_syn_backlog=4096
```
#### III. Увеличить лимит на используемые ресурсы для пользователя (на котором запущен сервер)

- Конфиг ослабляет лимиты на используемые файлы пользователем minecraft, такие как файлы/чанки/процессы
- Полезен. Без этой фичи при 50+ игроках и большоим количестве плагинов часто вылетает ошибка в консоли "Too many open files", она же ошибка "java.io.IOException: Too many open files"

**Ввожу команду:**
	``
	` sudo nano /etc/security/limits.conf`

**Добавляю в конфиг:**
```
minecraft soft nofile 65535
minecraft hard nofile 65535
```

Залогиниваюсь в пользователя "minecraft" и проверяю командами:
	`su - minecraft`
	`ulimit -n`

- *(должно быть 65535*)

Разлогиниваюсь:
	`exit`


#### IV.  Настройка планировщика ввода-вывода NVMe-диска (опционально, пропускать на VPS с виртуальными дисками)

- Простыми словами: любой процесс когда хочет прочитать/записать данные на диск - добавляется в очередь. Отключив ее для нашего диска получится отправлять их напрямую на диск и убрать очередь ожидания, так как на современных NVMe уже есть контроллер и драйвер, который распределяет запросы и т.п.
- Не делается на хостингах с виртуальными дисками. Не имеет смысла.

**Ввожу команды:

- Просмотр имени устройства (обычно nvme0n1):
	`nvme list`
	
	*Вывод будет примерно такой:*
	*/dev/nvme0 /dev/nvme0n1*

- Посмотреть текущий планировщик^
	`cat /sys/block/nvme0n1/queue/scheduler`
	
	*P.S. Заменить `nvme0n1` на своё имя, если отличается*

#### Чтобы настройка **I/O scheduler = none** не слетала после перезагрузки сервера - создаем **udev-правило**:

1. **Создай файл правила** (самый простой и универсальный вариант для NVMe):
	`sudo nano /etc/udev/rules.d/60-nvme-scheduler.rules`
	
	- Вставь туда ровно это (одна строка):
		`ACTION=="add|change", KERNEL=="nvme[0-9]*n[0-9]*", ATTR{queue/scheduler}="none"`
		
	- Сохрани (Ctrl+O → Enter → Ctrl+X).
	- Если хочешь строго для nvme0n1 - замени на `KERNEL=="nvme0n1"`
	- Перезагрузи систему `reboot`

2. **Проверь, применилось ли, командой**:
	`cat /sys/block/nvme0n1/queue/scheduler`
	
	*Должно показать `[none]` в скобках*

Файлы в /home/minecraft/server.

### 7.1: Скрипт Запуска (start.sh с Aikar's Flags)
Адаптировано для 16 ГБ: Xmx10G (место под буфер для zRAM/OS).

1. Переходим в папку сервера
```
cd /home/minecraft/server
```
2. Создаём (или перезаписываем) файл start.sh через nano
```
nano start.sh
```


**Вставьте следующий код (подставьте своё имя jar-файла):**
```
#!/bin/bash
exec > /home/minecraft/server/start-debug.log 2>&1
set -x    # ← покажет ВСЕ выполняемые команды
echo "DEBUG: Script started as $(whoami) in $(pwd)"
echo "DEBUG: SCREENDIR=$SCREENDIR  HOME=$HOME"
# -------------------------------
# Запуск Minecraft сервера в screen с Aikar's flags
# -------------------------------

set -e  # Выходить при любой ошибке

cd /home/minecraft/server || exit 1

# Настройки
JAR="purpur-1.21.11-2566.jar"
MAXRAM="10G"
MINRAM="$MAXRAM"
DELAY=30
SCREEN_NAME="minecraft"
JAVA_PATH="/usr/bin/java"

FLAGS_AIKAR="-XX:+UseG1GC -XX:+ParallelRefProcEnabled -XX:MaxGCPauseMillis=200 \
    -XX:+UnlockExperimentalVMOptions -XX:+DisableExplicitGC -XX:+AlwaysPreTouch \
    -XX:G1NewSizePercent=30 -XX:G1MaxNewSizePercent=40 -XX:G1HeapRegionSize=8M \
    -XX:G1ReservePercent=20 -XX:G1HeapWastePercent=5 -XX:G1MixedGCCountTarget=4 \
    -XX:InitiatingHeapOccupancyPercent=15 -XX:G1MixedGCLiveThresholdPercent=90 \
    -XX:G1RSetUpdatingPauseTimePercent=5 -XX:SurvivorRatio=32 \
    -XX:+PerfDisableSharedMem -XX:MaxTenuringThreshold=1 \
    -Dusing.aikars.flags=https://mcflags.emc.gs -Daikars.new.flags=true"

while true; do
    echo "=== [$(date '+%Y-%m-%d %H:%M:%S')] ЗАПУСК СЕРВЕРА ===" >> restart.log
    
    # Запускаем сервер в screen
    screen -dmS "$SCREEN_NAME" \
        "$JAVA_PATH" -Xms${MINRAM} -Xmx${MAXRAM} \
        ${FLAGS_AIKAR} \
        -jar "${JAR}" nogui

    echo "=== [$(date '+%Y-%m-%d %H:%M:%S')] Сервер запущен в screen ($SCREEN_NAME). Код возврата: $? ===" >> restart.log

    # Даём время на запуск и создание сокета
    sleep 15

    # Ждём, пока сессия жива
    while screen -list | grep -q "$SCREEN_NAME"; do
        sleep 10
    done

    echo "=== [$(date '+%Y-%m-%d %H:%M:%S')] СЕРВЕР ОСТАНОВЛЕН. Перезапуск через ${DELAY} сек... ===" >> restart.log
    echo "Для отмены авторестарта нажмите Enter в течение ${DELAY} секунд..."

    read -t $DELAY || true
    if [ $? -eq 0 ]; then
        echo "=== [$(date '+%Y-%m-%d %H:%M:%S')] Авторестарт отменён ===" >> restart.log
        break
    fi
done
```

**Сохраните и сделайте исполняемым:**
```
chmod +x start.sh
```

### 7.3: Автозапуск через systemd (рекомендуемый способ)

#### Создаем нужные скрипты

1. Если залогинены в пользователя minecraft - выходим командой `exit`

2. Создаем директорию вручную и даем правильные права:
```
sudo mkdir -p /home/minecraft/.screen
sudo chown minecraft:minecraft /home/minecraft/.screen
sudo chmod 700 /home/minecraft/.screen
```

2. Создаём юнит-файл от root:
```
sudo nano /etc/systemd/system/minecraft.service
```

- *Содержимое:*
```
[Unit]
Description=Minecraft Server (screen + Aikar's flags)
After=network.target

[Service]
Type=simple
User=minecraft
Group=minecraft
WorkingDirectory=/home/minecraft/server
Environment="HOME=/home/minecraft"

# Запускаем наш скрипт
ExecStart=/bin/bash /home/minecraft/server/start.sh

# Важные параметры
Restart=on-failure
RestartSec=12
TimeoutStopSec=90
KillSignal=SIGTERM
LimitNOFILE=65535
Nice=5
PrivateTmp=true
ProtectSystem=full
ReadWritePaths=/home/minecraft

[Install]
WantedBy=multi-user.target

```

- *Применяем изменения:*
```
sudo systemctl daemon-reload
sudo systemctl enable minecraft
sudo systemctl start minecraft
```

- *Проверка статуса:*
```
sudo systemctl status minecraft -l
```
